### Established, reusable patterns (follow these for every remaining migration)
1. **Reducer owns the state change + emits a thin Effect; the runner executes only the side effect.** This is the send/edit model (already in place for send/edit/interrupt/moveSessionTab/setPrefs).
2. **Hard cutover per op:** once the reducer owns it, delete the legacy Effect + Result event + reducer/result-handlers cases + service method + interface decls in the same chunk. The exhaustiveness `never` checks are the safety net — a dangling reference fails the build.
3. **Optimistic apply + revert-on-failure** (send/edit-style) for settings that can fail: snapshot the pre-change value in `PendingState` keyed by `corrId`; on `*Result{ok:false}` revert + notice; on `{ok:true}` clear the snapshot. (Used for send/edit.) **Not universal** — `setPruningSettings` (commit `fca20c7`) deliberately does NOT snapshot/revert: the service keeps graceful-degradation (catch+mirror+notice, no throw) so the result is always `{ok:true}`, and the reducer's optimistic apply is authoritative. Choose revert-on-failure only when the side effect can actually fail in a way the user must be shielded from (and pair it with pattern #7 — verify the disk-write covers every field).
4. **Effects are thin descriptors**, not fat service calls. If an op needs a UI dialog, model it as a dedicated modal Effect with a result event the reducer branches on — never do a modal inside an effect's service call. **Established by setModel (`6b84d2a`):** `ShowModelSwitchConfirm` Effect + `ModelSwitchConfirmResult` — the reducer stashes the intent (`snapshot:null`) and emits the modal effect with NO state change; `{confirmed}` applies + emits the RPC, `{!confirmed}` drops the intent unchanged (so a Cancel leaves everything untouched). The modal is intentionally NOT queued (VS Code serializes modals; the backend write stays queued) so it doesn't block session create/open the way the old `enqueueLifecycle(modal)` path did. **Two supporting moves** future modal ops should reuse: (i) **relocate session guards into the reducer** (setModel moved `requireOpenSessionPath` in) so an invalid request can't leave an optimistic change un-reverted; (ii) **extract pure helpers** the reducer needs but that live in vscode-importing modules into a pure-spine module (setModel extracted `modelSupportsInputKind` to `core/model-capability.ts` so the reducer doesn't pull vscode into its test graph).
5. **Persist via `PersistTabs` Effect** (now wired) rather than imperative `saveOpenTabs()` calls — migrate each tab op to emit `PersistTabs` and drop its `saveOpenTabs()` call.
6. **Sub-agent rhythm that works:** scout (recon, precise file:line map) → worker (TIGHT, one well-bounded change — workers time out on multi-change design tasks) → reviewer (independent verification) → orchestrator commits only the extension/ files (never `settings.json`). For design-heavy chunks, the orchestrator does the design + core implementation directly and uses workers only for mechanical sub-tasks.
7. **Verify the disk-write covers every field when adding a reducer optimistic apply for a persisted setting.** The reducer's optimistic merge must match the disk-write merge field-for-field, else optimistic state ≠ persisted state (UI shows one value, disk/restart another — silent data loss). When lifting `setPruningSettings`, `writePruningSettings` had no `prepassTimeoutSec` handler (the field was read by `readPruningSettings` but never written), so the optimistic apply would have diverged from the disk; fixed in the same chunk. Grep the write handler for each persisted field before declaring optimistic==persisted.
8. **In-flight guard keyed by `corrId` (one-in-flight-per-session ops).** For ops that must serialize per session (transcript paging), the reducer owns a `Record<sessionPath, corrId>` flag: the Command handler blocks if the flag is set, else sets it = `cmd.corrId` + emits the Effect; the `*Result` clears it only when `event.corrId === flag[sp]` (a stale result from a superseded request — e.g. tab closed+reopened — must NOT clear the current flag); `SessionScopeCleared` clears it on tab close. This replaces a host-side `Set`/seq-counter pair: `corrId` is the request identity (consistent with send/edit `PendingOp`), so a separate monotonic seq counter is unnecessary. Because `dispatchArchEvent` is synchronous, the clear happens synchronously via the Result event (no timing regression vs an old `finally`). Used for transcriptPaging (`7ffec0d`); reusable for any future one-in-flight-per-session op.
9. **Host-local lifecycle bookkeeping that snapshots pre-reducer state must run BEFORE the reducer's optimistic change.** `createSession` (`9343b62`) keeps the selection token + 60s timer + `handleSelectionFailure` host-side in `SessionServiceState` (the reducer can't own a timer/counter), but `beginSelectionRequest` snapshots `previousActivePath = archState.sessions.activeSessionPath` for failure recovery. The reducer's `CreateSession` handler sets `activeSessionPath = pending` optimistically. So `beginSelectionRequest` MUST run before the Command dispatch (it does, in `service.createNewSession()`, before `dispatchArch`), else it snapshots the pending path and `handleSelectionFailure` restores the wrong tab on failure. The token then flows through Command→reducer→Effect→runner (the reducer passes it through unchanged; the runner uses it for the backend RPC). **Reusable for `openSession` (same `beginSelectionRequest` + `handleSelectionFailure` machinery): mint the token before the reducer activates the opened tab.** Pinned by `create-session-ordering.test.ts` + the e2e recovery test. When moving host-local bookkeeping that reads ArchState, audit what it snapshots and whether the reducer's optimistic change would corrupt the snapshot — if so, run the bookkeeping before the Command dispatch and flow the result through the spine.

- `backendReadyQueue: QueuedBackendReadySend[]` — sends to restored sessions before the backend is ready. Drains when `backendReady` transitions false→true (`drainBackendReadyQueue()` → re-dispatches each entry through `handleMessage`).
- `backendReadyQueueWatchdog: NodeJS.Timeout | null` — 30s timer. On fire: splices the queue, dispatches `OptimisticMessageRemoved` for each entry with a `localId`, dispatches `NoticeShown` ("Backend did not become ready within 30s. N queued message(s) dropped — please retry.").
- Both enqueue methods (`enqueuePendingSend`, `enqueueBackendReadySend`) dispatch `OptimisticMessageInserted` + `SessionNameDerived` directly (bypassing the reducer — these are already Event-union members the reducer processes, but the dispatch is imperative, not reducer-owned).
- Constructed at `message-router.ts:77`; callers are `message-router.ts` (enqueue from `onSend` when `isPendingTabPath` or `!backendReady`) + `extension-host.ts` (drain on pending-resolve + drain on backend-ready + clear watchdog on shutdown).
- **The `handleMessage` re-dispatch is the trickiest part:** drained sends re-enter `MessageRouter.handle()` → `onSend` → (this time the gating conditions are false) → actual backend dispatch. The `localId` is preserved so the optimistic message already in the transcript gets matched, not duplicated. Moving this into the reducer means the reducer can't re-dispatch Commands — it must emit Effects that the runner executes by re-dispatching the queued sends.

**Migration approach (hard cutover, TDD):**
1. Move `pendingSendQueue` + `backendReadyQueue` into `ArchState` as `Record<string, T>` (Record-only rule per `STATE_CONTRACT.md`). The `Send` Command handler in the reducer already runs for every send — when the session is pending or `!backendReady`, the reducer queues the send into the ArchState Record (instead of emitting `SendRpc`) and dispatches `OptimisticMessageInserted` + `SessionNameDerived` as Effects (or directly as state transitions — the reducer already owns transcript state).
2. The drain-on-pending-resolve becomes a reducer reaction to `PendingPathReplaced` (when a pending path resolves to a real path, the reducer emits `SendRpc` Effects for each queued entry, keyed by `corrId`). The drain-on-backend-ready becomes a reducer reaction to `BackendReadyChanged{ready:true}`.
3. The 30s watchdog becomes a timer Effect: the reducer emits a `StartWatchdog` Effect when a send is queued into `backendReadyQueue`; the runner starts a `setTimeout`; on fire, dispatches a `BackendReadyWatchdogFired` event → reducer dispatches `OptimisticMessageRemoved` + `NoticeShown` (or the reducer directly removes the optimistic messages + sets the notice, since it owns transcript state). Alternatively, keep the timer host-side in the runner (like the selection-token timer in `SessionServiceState`) and just move the queue data into ArchState.
4. Hard-delete `QueueManager` once all state + logic is in the reducer/runner. The `handleMessage` re-dispatch loop becomes the runner executing the emitted `SendRpc` Effects.

**Suggested first chunk:** start with `pendingSendQueue` only (simpler — no timer, drains on `PendingPathReplaced` which is already a reducer event). Move the queue data into `ArchState`, make the `Send` Command handler queue when `isPendingTabPath(sessionPath)`, make `handlePendingPathReplaced` drain (emit `SendRpc` Effects). Then do `backendReadyQueue` + the watchdog as a second chunk.

### Phase 4 — delete legacy `SessionService` orchestration
Once every op routes through Command→reducer→effect with the reducer owning transitions, strip the orchestration from `SessionService`; it becomes a thin backend adapter + lifecycle (backend client start/stop, session boot). Shrink `SessionServiceState` (`state.ts`, ~461 lines — holds epoch counters, selection requests, transcript-window LRU, suppress flags, busy seq). `runObserver` calls sprinkled across `handlers/*` + `message-actions.ts` + `tab-actions.ts` — decide whether the observer subscribes to ArchState projections or stays as Effect-side calls. The module-global `subscribeToArchState` auto-projection (`dispatch.ts` + `extension-host.ts`) can become an explicit `selectViewState` call after dispatch.

### Phase 5 — harden recovery + webview
- **Optimistic-op TTL:** if no `*Result` arrives within N seconds, reconcile against backend (query session) or revert + notify. Today optimistic messages + `pending.ops[corrId]` live forever if a result is lost (host crash mid-send). No escape hatch.
- **Transport recovery:** `sidebar/sync.ts` `reconcilePostedMessageDelivery` only marks `globalDirty` for `type==='state'` — a lost `sendRejected` (restores the user's draft) or imperative is silently dropped (data loss). Recover all state-bearing host→webview messages. `canPostSnapshotToWebview` ignores the `webviewReady` flag even though the provider tracks it — gate posting on ready. The webview message listener (`webview/panel/hooks/use-host-sync.ts`) doesn't verify `event.source`/origin.
- **Recovery-by-reload:** a missed `stateApplied` ack force-reloads the whole webview (`provider.ts`: `STATE_APPLIED_TIMEOUT_MS=2500`, `STATE_APPLIED_RELOAD_LIMIT=2`/30s) — a sledgehammer that can cause reload storms on slow transcripts. Replace with re-snapshot-first; reload only after consecutive unacked re-snapshots.
- **Webview state-ownership leaks (contract violations):** module-level `Map` disclosure cache (`webview/panel/transcript/use-disclosure-open.ts`) shadowing host `autoExpand*` prefs and never cleared on host-instance/session change (clear violation); local `dismissed` boolean for host-owned error content (`transcript/message-item/error-detail.tsx`); ~14 borderline disclosure locals; fragile composer draft re-sync keyed only on `[sessionPath]` (`composer/hooks.ts`); `pendingDraftRestoreRef` Map never cleared on session close (`use-host-sync.ts` `clearTransientUi`). Move disclosure/error-dismissal into host state (or at minimum clear the module Map on host-instance/session change); re-sync composer draft on host `draftText` change.
